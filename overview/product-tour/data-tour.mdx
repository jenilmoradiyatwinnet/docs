## Main data areas

<Tabs>
  <Tab title="Events">
    Events are the foundation of product analytics.

    | Field | Meaning |
    | --- | --- |
    | **Event Name** | Original event received from SDK or API |
    | **Custom Event Name** | User-friendly name shown in reports |
    | **Source** | Where the event came from |
    | **Auto Status** | Whether event is tracked automatically |
    | **Manual Status** | Whether event is manually configured |
    | **Visible** | Whether event appears in reports |

    <Tip>
      Keep event names clear and consistent so reports remain easy to understand.
    </Tip>
  </Tab>

  <Tab title="Properties">
    Properties describe extra information attached to data.

    <CardGroup cols={2}>
      <Card title="User properties" icon="user">
        Name, email, country, plan, signup date.
      </Card>

      <Card title="Event properties" icon="bolt">
        Product, price, category, screen name.
      </Card>

      <Card title="Session properties" icon="clock">
        Duration, source, campaign, entry screen.
      </Card>

      <Card title="Device properties" icon="mobile">
        OS, app version, browser, brand, network.
      </Card>
    </CardGroup>
  </Tab>

  <Tab title="Sources">
    Sources help identify where activity came from.

    <AccordionGroup>
      <Accordion title="App source">
        Events sent from Android, iOS, Flutter, or mobile SDKs.
      </Accordion>

      <Accordion title="Web source">
        Events sent from website or web SDK.
      </Accordion>

      <Accordion title="API source">
        Events sent from backend or server-side integrations.
      </Accordion>

      <Accordion title="Integration source">
        Events or data connected from tools such as Firebase, Shopify, or OneSignal.
      </Accordion>
    </AccordionGroup>
  </Tab>

    <Warning>

  </Tab>
</Tabs>

## Common workflow

<Steps>
  <Step title="Review incoming events">
    Check which events are being received from SDKs, APIs, or integrations.
  </Step>

  <Step title="Clean names">
    Add custom event names so teams can read reports easily.
  </Step>

  <Step title="Manage visibility">
    Hide noisy or unwanted events from reports when needed.
  </Step>

  <Step title="Validate quality">
    Check duplicate, missing, or invalid events before using reports for decisions.
  </Step>
</Steps>

## Best practices

<Check>Use clear names like <code>signup_completed</code> or <code>purchase_completed</code>.</Check>
<Check>Keep property names consistent across platforms.</Check>
<Check>Hide test events from reports.</Check>
<Check>Review new SDK events after every release.</Check>
